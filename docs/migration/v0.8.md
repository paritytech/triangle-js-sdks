# Migration Guide: v0.7 → v0.8

This guide covers all breaking changes and new APIs introduced in v0.8. It is split by SDK so each area can be migrated — and published — independently:

- [**host-api**](#host-api) — the host ↔ product wire protocol (`host-api`, `host-api-wrapper`)
- [**host-papp**](#host-papp) — the Polkadot Mobile integration layer (`host-papp`, `host-papp-react-ui`)
- [**Host libraries**](#host-libraries) — supporting runtimes (`host-worker-sandbox`)
- [**Common**](#common) — low-level pieces shared across SDKs (`@novasamatech/statement-store`)

> **v0.8 is wire-incompatible with v0.7.** A v0.8 host will not talk to a v0.7 product or a v0.7 Polkadot Mobile build, and vice-versa — there is no compatibility shim. Existing paired sessions must be re-paired; see [Multi-device SSO V2](#multi-device-sso-v2).

---

# host-api

The host ↔ product wire protocol (`host-api`, `host-api-wrapper`). These are **wire-level** changes — a host and a product must run compatible versions. There is no cross-version compatibility shim.

> **Why:** the canonical TrUAPI protocol specification and the `host-api` SDK had drifted apart in a few places. v0.8 reconciles them so the spec and the wire format the SDK speaks are identical again. Most of the reconciliation happened on the spec side and required no SDK change; the items below are the ones that *did* change the SDK and therefore affect you.

## TL;DR

1. **`StorageQueryItem` — rename the `type` field** (only if you hand-build chain storage queries; `createPapiProvider` needs no change).

   ```diff
   - { type: 'value', /* ... */ }
   + { queryType: 'value', /* ... */ }
   ```

2. **Variant tags renamed** (only if you hand-build payloads): `WebRTC`→`WebRtc`, `BulletInAllowance`→`BulletinAllowance`.
3. **Hand-rolled custom renderer** — rename design tokens (e.g. `titleXL`→`headline.large`, `textPrimary`→`fg.primary`); `@novasamatech/product-react-renderer` already maps them.
4. **Verify, don't edit:** the `OptionBool` bit-fix means you should re-test signing (`withSignedTransaction`) and custom-renderer round-trips against the peers you talk to. No source change on your side.
5. **Top-up secret keys** — the `coins` and `privateKey` top-up sources take 64-byte sr25519 secret keys; see [coin top-ups](#coin-top-ups).
6. **Theme subscription payload changed shape** — `host_theme_subscribe` now delivers a `{ name, variant }` struct instead of a flat `'light' | 'dark'`. Read `theme.variant` for the old light/dark value (now capitalized `'Light' | 'Dark'`).
7. **No action needed:** JSON-RPC stub methods removed (no real callers); the payment purse selector and the `coins` top-up source are additive.

---

## JSON-RPC methods removed

The long-deprecated `host_jsonrpc_message_send` and `host_jsonrpc_message_subscribe` methods are gone. They were stubs superseded by the typed `remote_chain_*` methods:

| Removed | Use instead |
|---|---|
| `host_jsonrpc_message_send`, `host_jsonrpc_message_subscribe` | `remote_chain_head_follow_subscribe`, `remote_chain_head_header` / `_body` / `_storage` / `_call` / `_unpin` / `_continue` / `_stop_operation`, `remote_chain_spec_genesis_hash` / `_chain_name` / `_properties`, `remote_chain_transaction_broadcast` / `_stop` |

If you reach a chain through `host-api-wrapper`'s `createPapiProvider`, you are already on `remote_chain_*` and need no code change.

The two methods occupied a contiguous block of protocol method ids. v0.8 keeps that block **reserved as a gap** rather than compacting it, so every surviving method keeps the exact id it had in v0.7 — removing the stubs does not shift the rest of the protocol. Only direct callers of the two methods are affected, and there were none in practice.

---

## `OptionBool` encoding fixed

`OptionBool` (the compact `Option<bool>` codec) encoded `Some(true)` / `Some(false)` with the bytes swapped relative to canonical SCALE. It now matches `parity_scale_codec`:

| value | old byte | new byte |
|---|---|---|
| `undefined` (None) | 0 | 0 |
| `true` | 2 | **1** |
| `false` | 1 | **2** |

You don't call the codec directly, but it backs two protocol fields:

- **signing** — `withSignedTransaction` on the sign-payload requests;
- **custom renderer** — `enabled` and `loading` on `ButtonProps`, `enabled` on `TextFieldProps`.

Any value encoded by an older build reads inverted on a v0.8 peer. Nothing in your own code changes — but re-verify signing and custom-renderer round-trips against the peers you actually talk to.

---

## Payment purse selector

Payment requests gained an optional purse selector (RFC-0017), so a product can target a specific purse instead of the main one. On the wire this adds an optional field to three requests; in the `host-api-wrapper` payment manager it is an optional trailing argument.

```ts
import { createPaymentManager } from '@novasamatech/host-api-wrapper';
import type { PurseId } from '@novasamatech/host-api-wrapper';

const payments = createPaymentManager();
const purse: PurseId = 7; // PurseId = number

// Before — always the main purse
payments.subscribeBalance(onBalance);
await payments.topUp(100n, source);
await payments.requestPayment(50n, destination);

// After — optional selector; omit for the main purse (behavior unchanged)
payments.subscribeBalance(onBalance, purse);
await payments.topUp(100n, source, /* into */ purse);
await payments.requestPayment(50n, destination, /* from */ purse);
```

Existing call sites that omit the selector keep working unchanged. Container handlers receive the selector as an extra field on the decoded request (`purse` on balance-subscribe, `into` on top-up, `from` on request) — existing handlers can ignore it.

## Coin top-ups

`topUp` gained a third source — `coins` (RFC 0021) — that credits a balance directly from raw sr25519 coin secret keys, with no on-chain round trip. It sits alongside the existing `productAccount` and `privateKey` sources.

```ts
await payments.topUp(75n, { type: 'coins', keys }); // keys: Uint8Array[]
```

The host validates each key, claims the coins, and credits the target purse; spent or sniped coins are skipped. When only some coins could be claimed, the call rejects with the new **`PaymentTopUpErr.PartialPayment`** error carrying `{ credited }` — the amount that was credited:

```ts
try {
  await payments.topUp(75n, { type: 'coins', keys });
} catch (e) {
  if (e.payload?.credited !== undefined) {
    // partial: e.payload.credited was added; the rest could not be claimed
  }
}
```

Container handlers receive the source as `{ tag: 'Coins', value: keys }`. Existing handlers that only match `ProductAccount` / `PrivateKey` are unaffected unless they need to accept coin top-ups.

> **Key format.** Both the per-coin `coins` keys and the `privateKey` source are **64-byte sr25519 secret keys**. The `host-api-wrapper` `TopUpSource` shape takes `Uint8Array` / `Uint8Array[]`, but each value must be a 64-byte sr25519 secret key.

## Theme subscription payload

`host_theme_subscribe` previously delivered a flat enum — `'light' | 'dark'`. It now delivers a struct carrying both the active theme **name** and its **variant**:

```
enum ThemeName    { Custom(string), Default }
enum ThemeVariant { Light, Dark }
struct Theme      { name: ThemeName, variant: ThemeVariant }
```

This is a **wire change** — the payload bytes and the decoded shape both differ. Note the variant casing also changed: `light`/`dark` → `Light`/`Dark`.

```diff
// host-api-wrapper: subscribeTheme callback
- subscribeTheme((mode) => {           // mode: 'light' | 'dark'
-   applyTheme(mode);
+ subscribeTheme((theme) => {          // theme: { name, variant }
+   applyTheme(theme.variant);         // 'Light' | 'Dark'
+   if (theme.name.tag === 'Custom') loadCustomTheme(theme.name.value);
  });
```

Host implementations of the container's `handleThemeSubscribe` handler must now emit the `Theme` struct instead of a `'light' | 'dark'` string. `host-api` exports `Theme`, `ThemeName`, and `ThemeVariant` for constructing and inspecting the payload.

## Renamed fields and variants

Pure renames — same wire bytes, updated TypeScript names:

| Where | Before | After |
|---|---|---|
| `StorageQueryItem` field | `type` | `queryType` |
| `RemotePermission` variant tag | `WebRTC` | `WebRtc` |
| `AllocatableResource` variant tag | `BulletInAllowance` | `BulletinAllowance` |

```diff
// RemotePermission variant tag
- { tag: 'WebRTC', value: undefined }
+ { tag: 'WebRtc', value: undefined }

// AllocatableResource variant tag (resource-allocation request)
- { tag: 'BulletInAllowance', value: undefined }
+ { tag: 'BulletinAllowance', value: undefined }
```

If you build chain storage queries by hand, rename the item field `type` → `queryType`. Going through `createPapiProvider` needs no change.

## Custom renderer design tokens

The custom chat renderer's design tokens were migrated to the new hierarchical design-system scale. Two SCALE enums in the custom-renderer protocol changed their variant **identifiers** — `TypographyStyle` and `ColorToken`. The variant **order is unchanged**, so the on-wire bytes are identical; this is a TypeScript-level rename. Any product that builds a custom renderer and hard-codes a token string must update it.

`TypographyStyle`:

| v0.7       | v0.8                   |
|------------|------------------------|
| `titleXL`  | `headline.large`       |
| `headline` | `title.medium.regular` |
| `bodyM`    | `body.large.regular`   |
| `bodyS`    | `body.medium.regular`  |
| `caption`  | `body.small.regular`   |

`ColorToken`:

| v0.7                  | v0.8                   |
|-----------------------|------------------------|
| `textPrimary`         | `fg.primary`           |
| `textSecondary`       | `fg.secondary`         |
| `textTertiary`        | `fg.tertiary`          |
| `backgroundPrimary`   | `bg.surface.main`      |
| `backgroundSecondary` | `bg.surface.container` |
| `backgroundTertiary`  | `bg.surface.nested`    |
| `success`             | `fg.success`           |
| `error`               | `fg.error`             |
| `warning`             | `fg.warning`           |

`ButtonVariant` (`primary` / `secondary` / `text`) and `ContentAlignment` are unchanged. If you render custom messages through `@novasamatech/product-react-renderer`, the package already maps the new tokens for you — only hand-rolled renderers need changes.

# host-papp

The Polkadot Mobile integration layer (`host-papp`, `host-papp-react-ui`).

## TL;DR

1. **`createPappAdapter` — drop the `metadata` URL, move host name/icon inline, rename the platform fields.**

   ```diff
   createPappAdapter({
     appId: 'my-host',
   - metadata: 'https://example.com/host-metadata.json',
     hostMetadata: {
   +   hostName: 'My Host',                              // was in host-metadata.json
       hostVersion: '1.2.3',
   +   hostIcon: 'https://example.com/icon-256.png',     // was in host-metadata.json
   -   osType: 'macOS',
   -   osVersion: '15.0',
   +   platformType: 'macOS',                            // renamed from osType
   +   platformVersion: '15.0',                          // renamed from osVersion
     },
   });
   ```

2. **Only if you hand-build chat payloads:** `chatAccepted` is now `{ messageId }` (was `undefined`).
3. **Only if you supply a custom `IdentityAdapter`:** implement `watchIdentity(accountId): Observable<Identity | null>`.

Everything else here is **additive** — device identity, `onAuthSuccess`, `StoredUserSession` fields, multi-device chat variants, live `watchIdentity`, the allowance service, and `abortPendingRequests()`. And remember the global rule: [upgrade host + mobile together and re-pair](#multi-device-sso-v2).

---

## Breaking changes

### Multi-device SSO V2

The SSO handshake has been replaced with the V2 multi-device protocol end-to-end. The consumer-facing surface on `pappAdapter.sso` — `pairingStatus`, `authenticate()`, `abortAuthentication()`, the `StoredUserSession` returned to consumers — is unchanged.

There is no V1 ↔ V2 compatibility:

- A host on v0.8 will not pair with a Polkadot Mobile build that only speaks V1.
- A host on v0.7 will not pair with a multi-device Polkadot Mobile build.
- Persisted v0.7 SSO session blobs decode short against the V2 codec and are silently dropped on first read — users with existing sessions need to re-pair.

Hosts and the paired Polkadot Mobile app must upgrade together.

---

### `createPappAdapter` parameter change

The `metadata: string` URL parameter (which pointed at a JSON file describing the host's `name` / `icon`) has been removed. Host identity now ships inline with the V2 QR proposal via the structured `hostMetadata` object, which also gains platform fields.

```ts
// Before — v0.7
createPappAdapter({
  appId: 'my-host',
  metadata: 'https://example.com/host-metadata.json',
  hostMetadata: {
    hostVersion: '1.2.3',
    osType: 'macOS',
    osVersion: '15.0',
  },
});

// After — v0.8
createPappAdapter({
  appId: 'my-host',
  hostMetadata: {
    hostName: 'My Host',
    hostVersion: '1.2.3',
    hostIcon: 'https://example.com/icon-256.png',
    platformType: 'macOS',
    platformVersion: '15.0',
  },
});
```

Hosts that previously served a `host-metadata.json` can drop the file — its contents now ride inline on `hostMetadata` as `hostName` and `hostIcon`.

---

### `HostMetadata` reshape

```ts
// Before — v0.7
type HostMetadata = {
  hostVersion?: string;
  osType?: string;
  osVersion?: string;
};

// After — v0.8
type HostMetadata = {
  hostName?: string;
  hostVersion?: string;
  hostIcon?: string;
  platformType?: string;
  platformVersion?: string;
  custom?: { name: string; value: string }[];
};
```

Mechanical renames:

| v0.7        | v0.8              |
|-------------|-------------------|
| `osType`    | `platformType`    |
| `osVersion` | `platformVersion` |

The new optional fields — `hostName`, `hostIcon`, `custom` — replace what used to be fetched from the `metadata` URL and add a place for host-specific extension entries.

---

### `MessageContent.chatAccepted` payload

The legacy single-device `chatAccepted` variant used to be a `_void` marker. It now carries the originating chat-request message id so the recipient can correlate the accept with its request.

```ts
// Before — v0.7
{ tag: 'chatAccepted', value: undefined }

// After — v0.8
{ tag: 'chatAccepted', value: { messageId: '...' } }
```

Older clients that emit the v0.7 `_void` shape will fail to decode against the v0.8 codec.

The new multi-device accept lives at a separate index (`deviceChatAccepted`, index 20) — see [Multi-device chat message variants](#multi-device-chat-message-variants) below.

---

### `IdentityAdapter.watchIdentity`

The `IdentityAdapter` interface — what you pass as `adapters.identities` to `createPappAdapter` — gains a required `watchIdentity` method alongside the existing `readIdentities`. The default RPC adapter already implements it; this only affects consumers who supply a custom identity adapter (testing harnesses, non-browser environments).

```ts
type IdentityAdapter = {
  readIdentities(accounts: string[]): ResultAsync<Record<string, Identity | null>, Error>;
  // NEW — emit the latest identity for one account and re-emit on change.
  // Error the stream when the underlying storage/pallet is unavailable.
  watchIdentity(accountId: string): Observable<Identity | null>;
};
```

A minimal adapter that doesn't track live updates can satisfy the contract by emitting once from `readIdentities` and completing — but emitting on chain changes is what powers the live [`useIdentity`](#live-identity-with-watchidentity) hook.

---

## New features

### Auto-persisted device identity

The V2 protocol identifies the host by a per-installation device identity (sr25519 statement-account key + X25519 encryption key). The SDK now creates and persists this identity for you on first run, against the configured `StorageAdapter`, and reuses it on subsequent launches — no consumer wiring needed for web hosts.

For Electron / native hosts that want a hardware-backed identity (Keychain, native secure storage), pass an optional `deviceIdentity` factory:

```ts
import type { DeviceIdentityForPairing } from '@novasamatech/host-papp';

createPappAdapter({
  appId: 'my-host',
  hostMetadata: { /* ... */ },
  async deviceIdentity(): Promise<DeviceIdentityForPairing> {
    return {
      statementAccountPublicKey,
      statementAccountSecret,
      encryptionPublicKey,
      encryptionPrivateKey,
    }
  },
});
```

The factory is invoked once per pairing flow.

### `onAuthSuccess` hook

A new optional caller hook fires after the V2 handshake succeeds — after the SDK has already written the session and secrets to its own repositories. Use it for consumer-specific post-pairing work (telemetry, custom peer caches, device-sync seeding).

```ts
createPappAdapter({
  appId: 'my-host',
  hostMetadata: { /* ... */ },
  onAuthSuccess: ({ session, identityChatPrivateKey }) => {
    // session: StoredUserSession (already persisted by the SDK)
    // identityChatPrivateKey: the private half of session.identityChatPublicKey;
    //   lives in UserSecretRepository and is surfaced here for consumers
    //   whose chat / device-sync layer needs it directly.
    telemetry.track('sso_paired', { sessionId: session.id });
  },
});
```

Throwing from the hook fails the pending `authenticate()` call.

### New `StoredUserSession` V2 fields

`StoredUserSession` gains two optional V2 fields, populated when pairing happens against a V2 mobile build:

```ts
type StoredUserSession = {
  // ... existing v0.7 fields ...
  identityAccountId?: AccountId;        // user's identity sr25519 account
  identityChatPublicKey?: Uint8Array;   // P-256 65-byte, derived locally from identityChatPrivateKey
};
```

Both are `Option`-wrapped on the wire so future schema revisions can append further fields without breaking decode of v0.8.0 blobs. The matching `identityChatPrivateKey` lives in `UserSecretRepository` and is surfaced through the [`onAuthSuccess`](#onauthsuccess-hook) hook.

### Multi-device chat message variants

`MessageContent` gains three new variants for the multi-device chat layer:

```ts
// Index 17 — announce a new peer device on the chat
{ tag: 'deviceAdded', value: { statementAccountId, encryptionPublicKey } }

// Index 18 — remove a peer device
{ tag: 'deviceRemoved', value: { statementAccountId } }

// Index 20 — multi-device accept.
// Sent on the identity-level session SessionId(B, A) encrypted with K(A, B)
// so all of A's devices can decrypt without per-device envelope.
{ tag: 'deviceChatAccepted', value: { requestId, device } }
```

Index 19 is reserved as a spec placeholder so `deviceChatAccepted` lands on its specified index.

### Live identity with `watchIdentity`

`pappAdapter.identity` gains `watchIdentity(accountId)` next to the existing one-shot `getIdentity`. It returns an RxJS `Observable<Identity | null>` that:

- emits the cached identity immediately (if one was persisted), then
- emits distinct on-chain values as they change, writing each non-null value through to storage, and
- falls back to `null` if the chain source stays silent past an internal timeout.

Concurrent watchers of the same account share a single chain subscription, which is torn down when the last subscriber unsubscribes.

```ts
const sub = pappAdapter.identity.watchIdentity(accountId).subscribe({
  next: (identity) => render(identity),   // null = no identity / not yet resolved
  error: () => render(null),              // source unavailable; re-subscribe to retry
});

// later
sub.unsubscribe();
```

The `useIdentity` hook in `@novasamatech/host-papp-react-ui` now uses this internally, so any component reading identity updates live as the user edits their on-chain profile — no code change needed on your side beyond upgrading. If you call `getIdentity` directly it still works unchanged.

See [`IdentityAdapter.watchIdentity`](#identityadapterwatchidentity) if you supply a custom identity adapter.

### Allowance service (`pappAdapter.allowance`)

`createPappAdapter` now returns an `allowance` service alongside `sso` / `sessions` / `secrets` / `identity`. It lets a product sign against a **host-managed allowance slot** — keyed by `(sessionId, productId)` — without nominating a product account. On first use for a slot it requests the key from the paired mobile app (RFC-0010 resource allocation) and caches it; later calls reuse the cached key. The slot is cleared when its session disconnects.

```ts
const pappAdapter = createPappAdapter({ appId, hostMetadata });

// Sign a Bulletin transaction against the allowance slot
const signerResult = await pappAdapter.allowance.getBulletinSigner(sessionId, productId);

// Create statement-store proofs against the allowance slot
const proverResult = await pappAdapter.allowance.getStatementStoreProver(sessionId, productId);

signerResult.match(
  (signer) => { /* PolkadotSigner */ },
  (err) => {
    // err: AllowanceError, err.reason ∈ 'NoSession' | 'Rejected' | 'NotAvailable' | 'UnexpectedResponse'
  },
);
```

This is purely additive — the new field doesn't affect existing `createPappAdapter` usage. `AllowanceService`, `AllowanceError`, and `AllowanceErrorReason` are exported from `@novasamatech/host-papp` for annotations and `instanceof` checks.

### Abort in-flight signing (`UserSession.abortPendingRequests()`)

`UserSession` (from `@novasamatech/host-papp`) gains an `abortPendingRequests()` method to cancel the remote-signing request that is in flight **and anything queued behind it** — e.g. when the user closes the signing UI before the paired mobile app responds. Each session has a single-slot request queue, so without it a stuck request would block the next `signPayload` / `signRaw` until the 180s queue timeout elapsed.

```ts
const task = userSession.signPayload(payload); // in flight

// User dismisses the signing sheet:
await userSession.abortPendingRequests(); // ResultAsync<void, Error>

// The in-flight `signPayload` (and any queued request) rejects, and the queue is free for the next request.
```

`abortPendingRequests()` drops the whole per-session request queue — the in-flight task and every request queued behind it are rejected immediately, freeing the single slot. It then delegates to [`Session.clearOutgoingBatch()`](#sessionclearoutgoingbatch) in `@novasamatech/statement-store` (see [Common](#common)).

This is **purely additive** — existing code that never calls `abortPendingRequests()` is unaffected.

# Host libraries

Supporting runtimes that aren't tied to a single SDK.

## TL;DR

`@novasamatech/host-worker-sandbox` can now run ES-module worker code — **opt-in** via a `resolveModule` hook. No code change if you don't use it, **but** every sandbox now loads the async (Asyncify) WASM, which is ~2× larger; budget for it if bundle size or cold-start matters. See below.

---

## ES module imports in the sandbox

`@novasamatech/host-worker-sandbox` can now run worker code that uses ES modules — static `import` / `export` and dynamic `import()`. It is **opt-in**: pass a `resolveModule` hook to `createSandbox` and the sandbox asks it for each module's source on demand. Omit the hook and behavior is exactly as before — an `import` declaration in worker code fails at load time.

```ts
type ResolvedModule = { filename: string; content: string | Uint8Array };

type ModuleResolver = (
  specifier: string,
  importer: string | null,
  defaultResolve: (specifier: string, importer: string | null) => string,
) => Promise<ResolvedModule | null> | ResolvedModule | null;
```

The hook is called for every import the VM encounters — the entrypoint's imports, transitive imports, and dynamic `import()`. The contract:

- **`importer`** is the resolved `filename` of the module issuing the import, or `null` for the entrypoint's own imports.
- **The resolver chooses each module's canonical `filename`.** Returning the **same `filename`** for two different specifiers makes them resolve to one shared module instance — it is parsed and executed once (dedup).
- **`content`** may be a `string` or a `Uint8Array` (decoded as UTF-8).
- Returning **`null`** signals "not found"; the import fails with `Module not found: <importer> -> <specifier>`. **Throwing or rejecting** surfaces that error to the import.
- **`defaultResolve`** POSIX-joins a relative specifier (`./x`, `../x`) against the importer's directory and collapses `.` / `..`; bare and absolute specifiers pass through unchanged. The common case is simply `(specifier, importer, defaultResolve) => archive[defaultResolve(specifier, importer)]`.

When `resolveModule` is configured, `run()` requires a `name` identifying the entrypoint so its relative imports resolve against it. Without `resolveModule`, `run(code)` is unchanged and `name` is ignored.

```ts
import { createSandbox } from '@novasamatech/host-worker-sandbox';

const archive: Record<string, string> = {
  'app/util.js': 'export const tag = 7;',
};

const sandbox = await createSandbox('example-product.dot', {
  resolveModule: (specifier, importer, defaultResolve) => {
    const filename = defaultResolve(specifier, importer);
    const content = archive[filename];
    return content === undefined ? null : { filename, content };
  },
});

await sandbox.run(
  `
    import { tag } from './util.js';
    __HOST_API_PORT__.postMessage(new Uint8Array([tag]));
  `,
  { name: 'app/index.js' },
);
```

`ModuleResolver`, `ResolvedModule`, `DefaultResolve`, and `ModuleId` are exported from `@novasamatech/host-worker-sandbox`.

> **Runtime change affecting every sandbox.** Supporting an async resolver requires QuickJS's Asyncify build, so `createSandbox` now always instantiates the async WASM variant (`newQuickJSAsyncWASMModule`) and evaluates the entrypoint with `evalCodeAsync` — whether or not `resolveModule` is set. This is **not** an API break: existing `createSandbox(id)` / `run(code)` call sites keep working untouched. But the Asyncify WASM is roughly **2× larger** than the previous synchronous binary and adds a small per-call overhead on every VM↔host boundary crossing (timers, `fetch`, `crypto`, messaging). All injected globals behave identically. If sandbox bundle size or cold-start matters for your host, budget for the larger WASM.

# Common

Low-level pieces in `@novasamatech/statement-store`, shared across SDKs. Both are additive — opt in when you need them.

## `Session.clearOutgoingBatch()`

`Session` in `@novasamatech/statement-store` gains a `clearOutgoingBatch()` method. It replaces the in-flight outgoing request batch with an **empty** statement on the same request channel, reusing the session's current expiry — the statement store keeps one statement per channel and rejects only a *strictly lower* expiry, so an equal expiry supersedes the live batch. Local outgoing state is always dropped and every pending response waiter is rejected (with `Error('Outgoing batch aborted')`), including messages still queued during session initialization and even if the superseding submission itself fails — in which case the submission error is surfaced to the caller.

This is **purely additive**. Existing code is unaffected; only consumers that supply a custom `Session` implementation (rather than using `createSession`) need to add `clearOutgoingBatch()` to satisfy the `Session` type. It backs the host-papp [`UserSession.abortPendingRequests()`](#abort-in-flight-signing-usersessionabortpendingrequests) method.

## Substrate slot-account crypto

`@novasamatech/statement-store` gains helpers for the mobile app's substrate slot-account signing scheme (`SecretKey::from_bytes`, used for host-managed allowance slots), backed by a new published WASM package `@novasamatech/substrate-slot-sr25519-wasm`:

```ts
import {
  ensureSubstrateSlotSr25519Ready,
  deriveSlotAccountPublicKey,
  signSlotAccountSecret,
  verifySlotAccountSignature,
} from '@novasamatech/statement-store';

await ensureSubstrateSlotSr25519Ready(); // load the WASM once, e.g. at host startup
const publicKey = deriveSlotAccountPublicKey(slotSecret);
```

This is plumbing for the multi-device allowance flow; most hosts won't call it directly. The pre-existing `deriveSr25519PublicKey` / `signWithSr25519Secret` / `verifySr25519Signature` keep the same signatures and self-initialize their WASM lazily, so existing call sites need no change.
