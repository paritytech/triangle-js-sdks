# Migration Guide: v0.6 → v0.7

This guide covers all breaking changes and new APIs introduced in v0.7, organized by feature area.

## Overview of Breaking Changes

- [Legacy Account API Renamed](#legacy-account-api-renamed) — all `*_non_product_account` identifiers renamed to `*_legacy_account` across wire protocol, container, and product-sdk
- [Signing Methods: `address` → `ProductAccountId`](#signing-methods-address--productaccountid) — `host_sign_raw` and `host_sign_payload` now require a typed account tuple; legacy string-based signing moved to `*_with_legacy_account` methods
- [Permission Model](#permission-model-rfc-0002) — `DevicePermissionRequest` → `DevicePermission` (new variants); `RemotePermissionRequest` → `RemotePermission` (restructured variants); `remote_permission` now accepts a batch
- [Statement Store Subscribe API](#statement-store-subscribe-api-rfc-0008) — start payload changed from `Vec<Topic>` to `TopicFilter`; receive payload changed from `Vec<SignedStatement>` to `SignedStatementsPage`
- [polkadot-api v2.0](#polkadot-api-v20) — `JsonRpcProvider` import path changed; WS provider path changed

---

## Breaking Changes

### Legacy Account API Renamed

All "non-product account" identifiers have been renamed to "legacy account" across every layer of the SDK. This is a
mechanical find-and-replace with no behavioral change.

#### Protocol (`host-api`)

| v0.6                                               | v0.7                                          |
|----------------------------------------------------|-----------------------------------------------|
| `host_get_non_product_accounts`                    | `host_get_legacy_accounts`                    |
| `host_create_transaction_with_non_product_account` | `host_create_transaction_with_legacy_account` |
| `host_sign_raw_with_non_product_account`           | `host_sign_raw_with_legacy_account`           |
| `host_sign_payload_with_non_product_account`       | `host_sign_payload_with_legacy_account`       |

#### Container handlers (`host-container`)

| v0.6                                           | v0.7                                       |
|------------------------------------------------|--------------------------------------------|
| `handleGetNonProductAccounts`                  | `handleGetLegacyAccounts`                  |
| `handleCreateTransactionWithNonProductAccount` | `handleCreateTransactionWithLegacyAccount` |
| `handleSignRawWithNonProductAccount`           | `handleSignRawWithLegacyAccount`           |
| `handleSignPayloadWithNonProductAccount`       | `handleSignPayloadWithLegacyAccount`       |

#### Product SDK (`product-sdk`)

| v0.6                                     | v0.7                                 |
|------------------------------------------|--------------------------------------|
| `createNonProductExtensionEnableFactory` | `createLegacyExtensionEnableFactory` |
| `getNonProductAccounts`                  | `getLegacyAccounts`                  |
| `getNonProductAccountSigner`             | `getLegacyAccountSigner`             |

### Signing Methods: `address` → `ProductAccountId`

`host_sign_raw` and `host_sign_payload` now identify the signer as a `ProductAccountId` instead of a bare address string. The payload field was also renamed from `data` to `payload`.

The old string-based signing path still works — it moved to the `*_with_legacy_account` methods.

**Before:**

```ts
// host_sign_raw
{ address: '5GrwvaEF...', data: { tag: 'Bytes', value: bytes } }

// host_sign_payload
{ address: '5GrwvaEF...', blockHash: '0x...', genesisHash: '0x...', ... }
```

**After (product account):**

```ts
// host_sign_raw — account is [dotNsIdentifier, derivationIndex]
{ account: ['my-app.dot', 0], payload: { tag: 'Bytes', value: bytes } }

// host_sign_payload
{ account: ['my-app.dot', 0], payload: { blockHash: '0x...', genesisHash: '0x...', ... } }
```

**After (legacy/external account — use the new legacy methods):**

```ts
// host_sign_raw_with_legacy_account
{ signer: '5GrwvaEF...', payload: { tag: 'Bytes', value: bytes } }

// host_sign_payload_with_legacy_account
{ signer: '5GrwvaEF...', payload: { blockHash: '0x...', genesisHash: '0x...', ... } }
```

For container implementors, wire up the new handlers:

```ts
container.handleSignRaw(({ account, payload }, { ok, err }) => {
  /* product account */
});
container.handleSignRawWithLegacyAccount(({ signer, payload }, { ok, err }) => {
  /* legacy account */
});

container.handleSignPayload(({ account, payload }, { ok, err }) => {
  /* product account */
});
container.handleSignPayloadWithLegacyAccount(({ signer, payload }, { ok, err }) => {
  /* legacy account */
});
```

### Permission Model

> Full spec: [RFC-0002 — Permission Model for Host API](../rfcs/0002-permission-model.md)

Both permission types were renamed and restructured.

#### `DevicePermissionRequest` → `DevicePermission`

The type is renamed and five new variants are added. Wire encoding of the original four variants is unchanged.

```ts
// Before — 4 variants
import { DevicePermissionRequest } from '@novasamatech/host-api';
// 'Camera' | 'Microphone' | 'Bluetooth' | 'Location'

// After — 9 variants
import { DevicePermission } from '@novasamatech/host-api';
// 'Notifications' | 'Camera' | 'Microphone' | 'Bluetooth'
// | 'NFC' | 'Location' | 'Clipboard' | 'OpenUrl' | 'Biometrics'
```

#### `RemotePermissionRequest` → `RemotePermission`

The variants are restructured. `ExternalRequest` and `TransactionSubmit` no longer exist.

```ts
// Before
import { RemotePermissionRequest } from '@novasamatech/host-api';
// { tag: 'ExternalRequest'; value: string }  — single URL
// { tag: 'TransactionSubmit' }

// After
import { RemotePermission } from '@novasamatech/host-api';
// { tag: 'Remote'; value: string[] }    — domain patterns (e.g. ['api.example.com', '*.cdn.io'])
// { tag: 'WebRTC' }
// { tag: 'ChainSubmit' }
// { tag: 'PreimageSubmit' }
// { tag: 'StatementSubmit' }
```

Domain pattern rules for `Remote`:

- `"api.example.com"` — exact match
- `"*.example.com"` — single subdomain wildcard
- `"*"` — all HTTP/WS traffic (shown with a prominent host warning)

#### `remote_permission` now accepts a batch

The request payload changed from a single permission to an array, so all remote needs can be declared in one prompt.

```ts
// Before — single permission
hostApi.permission(enumValue('v1', { tag: 'ExternalRequest', value: 'https://api.example.com' }));

// After — array of permissions
hostApi.permission(
  enumValue('v1', [
    { tag: 'Remote', value: ['api.example.com'] },
    { tag: 'ChainSubmit', value: undefined },
  ]),
);
```

#### Product SDK helpers

```ts
import { requestDevicePermission, requestPermission } from '@novasamatech/product-sdk';
import type { DevicePermissionKind, RemotePermissionItem } from '@novasamatech/product-sdk';

// Single device permission
const granted = await requestDevicePermission({ tag: 'Camera', value: undefined });

// Batch remote permissions
const allGranted = await requestPermission([
  { tag: 'Remote', value: ['api.coingecko.com'] },
  { tag: 'StatementSubmit', value: undefined },
]);
```

### Statement Store Subscribe API

> Full spec: [RFC-0008 — Statement Store Host API v0.2](../rfcs/0008-statement-store.md)

The subscribe payload changed in both directions to expose expressive filtering and paged delivery.

#### Start payload: `Vec<Topic>` → `TopicFilter`

`TopicFilter` supports two modes:

- `MatchAll` — statement must contain **every** listed topic (AND)
- `MatchAny` — statement must contain **at least one** listed topic (OR)

#### Receive payload: `Vec<SignedStatement>` → `SignedStatementsPage`

`SignedStatementsPage` adds an `isComplete` flag that signals the boundary between the initial historical dump and live
updates.

#### Product SDK (`product-sdk`)

```ts
// Before
createStatementStore().subscribe(
  [topic1, topic2],                     // Topic[]
  (statements) => {                     // SignedStatement[]
    render(statements)
  }
);

// After
createStatementStore().subscribe(
  { matchAll: [topic1, topic2] },       // StatementTopicFilter
  (page) => {                           // StatementsPage
    buffer(page.statements)
    if (page.isComplete) {
      render(buffer)  // initial dump complete; subsequent pages carry live updates
    }
  }
);

// OR semantics (new)
createStatementStore().subscribe({ matchAny: [topic1, topic2] }, ...);
```

#### Statement store adapter (`statement-store`)

```ts
// Before
adapter.queryStatements([topic1, topic2], destination);
adapter.subscribeStatements([topic1, topic2], statements => {});

// After
adapter.queryStatements({ matchAll: [topic1, topic2] }, destination);
adapter.subscribeStatements({ matchAll: [topic1, topic2] }, page => {
  // page.statements: Statement[]
  // page.isComplete: boolean
});
```

## New Features

### Payments API

> Full spec: [RFC-0006 — Payment Host API](../rfcs/0006-payments.md)

A new set of host API methods allows products to interact with the user's payment balance without knowledge of the
underlying settlement mechanism.

#### Product SDK

```ts
import { createPaymentManager } from '@novasamatech/product-sdk';
import type { PaymentBalance, PaymentStatus, TopUpSource } from '@novasamatech/product-sdk';

const payments = createPaymentManager();

// Subscribe to the user's spendable balance (requires user consent on first call)
const sub = payments.subscribeBalance(balance => {
  console.log(balance.available);
});
sub.unsubscribe();

// Top up the user's balance from a product-controlled account
await payments.topUp(100n, {
  type: 'productAccount',
  dotNsIdentifier: 'my-app.dot',
  derivationIndex: 0,
});

// Request a payment from the user (prompts for confirmation)
const { id } = await payments.requestPayment(50n, destination);

// Track payment until terminal state
const statusSub = payments.subscribePaymentStatus(id, (status: PaymentStatus) => {
  if (status.type === 'completed') {
    /* done */
  }
  if (status.type === 'failed') {
    console.error(status.reason);
  }
});
```

#### Container handler slots

```ts
container.handlePaymentBalanceSubscribe((_params, send, interrupt) => { });
container.handlePaymentTopUp(({ amount, source }, { ok, err }) => { });
container.handlePaymentRequest(({ amount, destination }, { ok, err }) => { });
container.handlePaymentStatusSubscribe((paymentId, send, interrupt) => { });
```

### Entropy Derivation

> Full spec: RFC-0007 — Deterministic Entropy Derivation for Products

`host_derive_entropy` derives deterministic 32-byte entropy scoped to the calling product and a caller-chosen key. The
same inputs always return the same bytes on any conforming host — useful for deriving stable key pairs (e.g. X25519)
without product-side key storage.

#### Product SDK

```ts
import { deriveEntropy } from '@novasamatech/product-sdk';

// key: up to 32 bytes, caller-defined purpose label
const result = await deriveEntropy(new TextEncoder().encode('x25519-keypair'));
// result: ResultAsync<Uint8Array /* 32 bytes */, DeriveEntropyErr>
```

#### Container

```ts
container.handleDeriveEntropy((key, { ok, err }) => {
  // implement derivation using RFC-0007 algorithm
  return ok(derive(key));
});
```

A `deriveProductEntropy` helper is also exported from `host-container` for reference implementations.

### Theme Subscription

Products can now subscribe to the host's current UI theme.

```ts
import { createThemeProvider } from '@novasamatech/product-sdk';
import type { ThemeMode } from '@novasamatech/product-sdk';

const theme = createThemeProvider();

const sub = theme.subscribeTheme((mode: ThemeMode) => {
  // mode: 'Light' | 'Dark'
  document.documentElement.dataset.theme = mode.tag.toLowerCase();
});

sub.unsubscribe();
```

Container handler:

```ts
container.handleThemeSubscribe((_params, send, interrupt) => {
  const stop = watchSystemTheme(theme => send(theme));
  return stop;
});
```

### Requesting root account from product

> Full spec: [RFC-0010 — Host API root account access](../rfcs/0010-get-root-account.md)

Returns the user's root (primary) account — the account the host associates with the user's DotNS identity. The host shows a permission prompt on the first call; the grant may be cached for the session.

#### Product SDK

```ts
import { createAccountsProvider } from '@novasamatech/product-sdk';
import { RequestCredentialsErr } from '@novasamatech/host-api';

const accountsProvider = createAccountsProvider();

const result = await accountsProvider.getRootAccount();

result.match(
  account => {
    console.log('Root public key:', account.publicKey);
    console.log('Root DotNS name:', account.name);
  },
  err => {
    if (err instanceof RequestCredentialsErr.Rejected) {
      // User denied — do not re-prompt in this session
    } else if (err instanceof RequestCredentialsErr.NotConnected) {
      // User has no DotNS username — fall back to product-scoped identity
    } else {
      console.error('Unexpected error:', err.value.reason);
    }
  },
);
```

#### Container handler slot

```ts
import { RequestCredentialsErr } from '@novasamatech/host-api';

container.handleAccountGetRoot(async (_, { ok, err }) => {
  const granted = await promptUserForRootAccountAccess();
  if (!granted) {
    return err(new RequestCredentialsErr.Rejected());
  }
  const rootAccount = await getRootAccount();
  if (!rootAccount) {
    return err(new RequestCredentialsErr.NotConnected());
  }
  return ok({ publicKey: rootAccount.publicKey, name: rootAccount.name ?? undefined });
});
```

### New: `host_request_login` (RFC-0009)

> Full spec: [RFC-0009 — Unauthenticated Product Access](../rfcs/0009-unauthenticated-product-access.md)

Products can now explicitly trigger the host login flow. The host shows its native sign-in UI and returns once the flow completes.

#### Product SDK

```ts
import { createAccountsProvider } from '@novasamatech/product-sdk';

const accountsProvider = createAccountsProvider();

accountsProvider.subscribeAccountConnectionStatus(status => {
  if (status === 'disconnected') {
    renderReadOnlyView({
      onLoginClick: async () => {
        const result = await accountsProvider.requestLogin('Sign in to vote on Referendum #42');
        result.match(
          loginResult => {
            if (loginResult === 'rejected') {
              // user cancelled — stay in read-only mode
            }
            // on 'success', the connection status subscription emits 'connected'
          },
          err => console.error('Login failed:', err),
        );
      },
    });
  }
});
```

#### Container handler slot

```ts
import { LoginErr } from '@novasamatech/host-api';

container.handleRequestLogin(async (reason, { ok, err }) => {
  const alreadyConnected = await checkIfConnected();
  if (alreadyConnected) return ok('alreadyConnected');

  const result = await presentLoginUI(reason);
  if (!result.success) return ok('rejected');

  return ok('success');
});
```

The default behaviour (before `handleRequestLogin` is wired up) returns `LoginErr.Unknown` with reason `"Not implemented"`.
